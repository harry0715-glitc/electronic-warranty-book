## 一、現況架構摘要

```
客戶端                    管理端                     後端
─────────                ─────────                 ─────────
index.html?token=  ←──   admin.html（建立）   ──→   Apps Script (Code.gs)
repair.html?token= ←──   query.html（查詢）   ──→   Google Sheet
LINE OA / LIFF     ←──   portal.html（入口）        ├ Warranties
                                                    ├ LineContacts
                                                    ├ Repairs
                                                    └ AdminUsers
```

- 前端：GitHub Pages 靜態頁（root 與 `docs/` 各一份）
- 後端：Apps Script 單檔 `Code.gs`（約 1,700 行 / 68KB），JSONP + GET API
- 資料：Google Sheet 四張工作表
- 通知：LINE OA webhook 綁定 + Flex 卡片推送

整體評價：MVP 架構合理、流程完整，token 化方向正確。主要風險集中在**安全收尾未完成**與**檔案重複維護**兩塊。

## 二、修改建議（依優先度排序）

### 🔴 P0 — 安全性（✅ 五項均已於 2026-07-10 修改完成，待部署）

**部署檢查清單（需由 Han 執行）：**

1. Apps Script Script Properties 設定強隨機 `ADMIN_API_KEY`（例如 32 字元以上亂數）——**未設定前管理金鑰驗證會停用，管理頁只剩 Google 帳號路徑可用**
2. 更換 `LINE_ADMIN_SEND_PIN` 與 `LINE_WEBHOOK_KEY`（舊 PIN `521527` 已在公開 repo 歷史中，視同外洩）；LINE Developers 後台的 webhook URL 同步改用新 key
3. `clasp push` → 建新版本 → redeploy 現行 Web App deployment
4. `git add -A && git commit && git push`（docs/ 已移除 admin.html、query.html）
5. GitHub repo Settings → Pages → source 切到 `main /docs`
6. 部署後 smoke test：後台建立測試案件 → token 開保固頁 → 送一筆測試報修 → 刪除測試資料

**1. 硬編碼 PIN 外洩於公開 repo** ✅ 已處理 2026-07-10（已刪除常數與 fallback 鏈，`getAdminApiKey_` 只認 Script Properties 的 `ADMIN_API_KEY`）
`Code.gs` 第 64 行：`DEFAULT_ADMIN_SEND_PIN = '521527'`。
repo 在公開 GitHub 上，若 Script Properties 未設 `ADMIN_API_KEY`，系統會 fallback 到這組 PIN——等於管理金鑰公開。
→ 做法：Script Properties 確實設定強隨機 `ADMIN_API_KEY`，刪除程式碼中的預設 PIN 與 fallback 邏輯，並更換 `LINE_ADMIN_SEND_PIN`。

**2. `docs/` 仍包含 admin.html / query.html** ✅ 已處理 2026-07-10（已刪除；portal.html 改指向 Apps Script 內建管理頁，走 Google 帳號驗證）
2026-06 安全調整原定 `docs/` 只留客戶頁，實際上 `docs/admin.html`、`docs/query.html` 都還在，且與 root 完全相同。管理頁形同全網公開。
→ 做法：從 `docs/` 移除 `admin.html`、`query.html`，Pages source 切到 `main /docs`，root 管理頁改走本機 `.hermes-local-admin-server.js` 或 Apps Script `?page=admin` 內建頁。

**3. adminKey 以 GET query string 傳送** ✅ 已處理 2026-07-10（admin.html / query.html 改 POST fetch，adminKey 移入 body；Code.gs 新增 `routeAdminAction_` 供 GET/POST 共用。註：Apps Script 內建 AdminApp/QueryApp 頁仍走 JSONP GET，但受 Google 帳號驗證保護）
JSONP 架構下 `adminKey` 出現在 URL，會留在 Apps Script 執行紀錄與瀏覽器歷史。
→ 短期：管理端改用 `doPost`（fetch + CORS 已可行，Apps Script 支援）。長期見擴充方案。

**4. LINE webhook 驗證不足** ✅ 已處理 2026-07-10
`handleLineWebhook_` 只驗 URL `?key=`，且 `configuredKey` 未設定時直接放行所有請求。
註：Apps Script 的 `doPost` 讀不到 request header，技術上無法驗 `X-Line-Signature`（HMAC），因此以「強制 URL webhook key」為實務替代方案。
→ 已改：key 未設定或不符一律拒絕。請確認 Script Properties 的 `LINE_WEBHOOK_KEY` 為強隨機值。

**5. `createRepair` 無保護的公開寫入** ✅ 已處理 2026-07-10（新增 `createPublicRepair_`：非管理請求必須帶有效 token；CacheService 頻率限制單一 token 10 分鐘 5 筆、全站 1 小時 30 筆）
走 GET + JSONP、無 rate limit，任何人可批量灌垃圾報修。
→ 做法：至少要求有效 `publicToken` 才能建立報修（綁定案件驗證），並加簡單頻率限制（CacheService 記 IP/token 次數）。

### 🟡 P1 — 結構與維護性

**6. root 與 `docs/` 完全重複**
六個主要檔案 + 全部 assets 一字不差各一份（含字型約 30MB×2）。改一處忘另一處是遲早的事。
→ 做法：確立 `docs/` 為唯一發佈來源，root 只留開發用檔案；或寫一個簡單 sync 腳本單向複製。

**7. 字型過大，客戶頁載入慢**
LINE Seed TW 三個 woff2 共約 10.7MB，TTF 三個共約 19.7MB。客戶手機開保固頁要載入數 MB 字型。
→ 做法：TTF 不上線（移出發佈目錄）；woff2 做中文子集化（pyftsubset 取常用字 + 頁面實際用字，可壓到 300–800KB）。

**8. Code.gs 單檔 1,700 行**
→ 做法：clasp 支援多檔，可拆為 `config.gs`、`warranty.gs`、`repair.gs`、`line.gs`、`admin.gs`、`util.gs`，行為不變、維護容易。

**9. 文件與設定不同步**
- README 的 apiBase（AKfycbxh…）與 `api-config.js` 現行值（AKfycbyI…）不一致
- `companyPhone: "0900-000-000"` 仍是占位符
→ 做法：更新 README、填入真實電話。

**10. 雜項檔案散落 root**
`make-transparent-logo.ps1`、`.hermes-local-admin-server.js`、`line-flex-message.json`、`line-push-sample.json` 混在發佈檔中。
→ 做法：建 `tools/` 與 `samples/` 歸檔；補 `.gitignore`。

### 🟢 P2 — 品質保障

**11. 寫入流程尚無 smoke test**（檢測紀錄已指出）
→ 做法：寫一支 `tools/smoke-test.js`：建測試案件 → token 開保固頁 → 送測試報修 → 查詢確認 → 刪除測試資料。每次 redeploy 後跑一次。

**12. 無資料備份機制**
Google Sheet 是唯一資料主體，誤刪即失。
→ 做法：Apps Script time trigger 每日將 Sheet 匯出 JSON/CSV 到 Drive 備份資料夾，保留 30 天。

**13. admin.html 的 localStorage `warranty_cases` 本地快取**
與 Sheet 可能不同步，多裝置操作會看到舊資料。
→ 做法：確認是否仍需要；若只是草稿暫存就明確命名並加清除機制。

## 三、可擴充方案

### 近期（現有架構即可做，Apps Script + Sheet）

| 方案 | 說明 | 依賴 |
|---|---|---|
| **保固到期提醒** | time trigger 每日掃 `warrantyEnd`，到期前 30/7 天 LINE push 提醒客戶與管理端 | 已有 push 基礎，成本低 |
| **報修照片上傳** | repair.html 加圖片欄位，存 Google Drive，Sheet 記連結（`photoUploadStatus` 欄位已預留） | Drive API |
| **儀表板完善** | `DashboardApp.html` 已有 3KB 雛形：案件數、保固中/已到期、待處理報修統計 | 已有雛形 |
| **LINE 關鍵字自動回覆** | 已綁定客戶輸入「報修」「查保固」直接回對應連結（進度紀錄已列為候選） | webhook 已有 |
| **保固書 PDF 輸出** | 用 Apps Script 由 Doc 範本產生 PDF，供列印或附加 LINE 傳送 | DocumentService |
| **多角色權限完善** | `AdminUsers` sheet 已有 `role` 欄位，可區分 owner / manager / viewer 操作權限 | 已有基礎 |
| **修改歷史記錄** | 案件每次編輯寫入 `AuditLog` sheet（who / when / what），對客訴糾紛有保障 | 新增一張表 |

### 中期（流程自動化）

- **報修工單流**：報修 → 指派廠商 → 排程 → 完工回報 → 客戶確認，狀態機化（`Repairs.status` 已存在，擴充狀態即可）
- **客戶滿意度回饋**：報修完工通知附評分連結，寫回 Sheet
- **n8n 導入時機**：當出現「跨系統」需求（如同步到會計軟體、Email 通知、行事曆排程）再導入，目前維持單純是對的

### 長期（規模成長後）

- **資料庫遷移**：Sheet 在數百案件內夠用；破千筆或需要多人並發寫入時，遷移到 Supabase / Firebase，Apps Script API 層先抽象好（`getOrCreateSheet_` 已集中，遷移面不大）
- **自訂網域**：GitHub Pages 綁 `warranty.你的網域.tw`，保固書連結更專業
- **與機電估算系統串接**：工作區內的機電試算若產出案件資料，可直接帶入保固書建立頁，減少重複輸入
